# AIProvider 项目开发规则

本文件适用于 `AIProvider`。修改代码前必须先阅读并遵守。

## 一、项目边界与权威上下文

1. `AIProvider` 是当前独立正式项目，不得与 `GamePod`、`AIMaid` 混改。
2. 当前主要模块职责：
   - `AIProvider-front`：React/Vite Web 工作区、界面和临时交互状态。
   - `AIProvider-back`：Spring Boot/MyBatis/Flyway 后端，负责业务编排、数据库状态、权限校验和外部服务接入。
   - `AIProvider-quant`：Quant 领域 Maven 大模块，作为普通 jar 被 `AIProvider-back` 依赖，仍随同一个 Spring Boot 进程和同一个后端 Jar 部署；不独立运行、不创建第二个服务、不引入 Web/数据库/MyBatis/Flyway/交易所 SDK/Spring Boot 插件。
   - `ComfyUIAgent`：本机 Bridge，负责 ComfyUI 生命周期、本机文件、生成任务和其他本地能力。
   - `CCXTGateway`：行情与交易所适配能力。
   - `TwitterChromeExtension`：浏览器侧 Twitter/X 发布适配能力。
   - `server-configs`：生产连接与发布配置的权威来源。
3. `AIMaid` 的 Electron、C# CoreHost、固定主题和旧 WPF 迁移规则不适用于本项目。
4. 当前代码、最新 `master` 提交和真实运行日志高于历史文档；文档与代码不一致时，以当前事实为准，并在同次任务中修正文档。
5. 不把项目当 Demo，不写一次性补丁，不为了“先跑起来”破坏正式结构。
6. 量化交易在 `AIProvider-front` 中保持单一一级导航入口 `/quant`；策略、回测与参数实验、风控中心、账户与仓位、订单与成交、运行记录属于 Quant 页面内部二级工作区，由 `QuantWorkbench` 以本地状态切换、路径始终为 `/quant`，不得拆成多个全局一级导航项或独立前端项目，不得新增 `/quant/*` 路由或引入 React Router。

## 二、核心原则

1. 不猜。看不懂、拿不准或资料可能过期时，先查当前代码、日志、数据库、官方文档、源码或 Issue。
2. 严禁兜底降级、吞异常、静默跳过、伪造成功、默认成功，或用重试、延迟、假数据掩盖根因。
3. 不扩大任务范围，不顺手重构无关代码，不重复建立已有能力。
4. 页面存在、按钮存在、接口存在或代码能够编译，都不能单独证明业务已经接通。
5. 开发中发现原方案不成立、需求冲突或新的关键问题时，立即停止当前实现并说明，不得擅自更换方案。
6. ASR 只接免费的第三方 API；除非主人以后明确改变要求，不得引入或部署 Whisper、FunASR 等本地 ASR 模型。

## 三、开源复用

1. 本项目为个人非商用项目；实现新模块时优先参考和复用 GitHub 上成熟、活跃且与当前技术栈匹配的实现，不因授权文本检查反复暂停开发或讨论。
2. 不得为了复用引入整套无关项目、替换现有架构或添加与任务无关的基础设施。
3. 复用内容必须接入本项目现有接口、主题、数据模型、错误处理和测试门禁，禁止只复制代码而不完成真实业务连接。
4. 不删除第三方代码中原有的版权或许可证声明，不把密钥、示例账号、构建产物和无关资源带入仓库。

## 四、开发流程

1. 写代码前先给出简短方案：改什么、怎么改、影响哪些模块和文件。
2. 需求不明确或存在多种业务实现时，先给主人明确选项；未确认前不得开发。
3. 已经出现的问题，先复现并检查日志、数据和完整调用链，确认根因后再修改。
4. 修改公共接口、DTO、数据库结构、配置、路由或协议时，必须同步检查全部调用方。
5. 完成后只说明实际修改、实际验证和仍未验证的部分，不夸大，不生成无关文档。
6. 不得把问题重新交给 Codex“全面检查”或“自行调查”；任务发出前必须已经明确问题、证据、文件范围和修改方案。

## 五、业务状态与完整调用链

1. 服务器业务必须检查完整链路：

   `React 页面 → API 调用 → Spring Controller → Service → Repository/Mapper → MySQL 或外部服务 → 返回协议 → React 状态`

2. 本机能力必须检查完整链路：

   `React 页面 → 本机 Bridge 调用与令牌 → ComfyUIAgent → ComfyUI/本机文件/设备能力 → 返回结果`

3. 需要长期保存的业务状态以 `AIProvider-back` 和数据库为准；React 只保存输入草稿、筛选、Tab、弹窗和加载状态等临时 UI 状态。
4. `ComfyUIAgent` 可以维护本机运行状态和本机任务队列，但不得代替后端数据库成为服务器业务记录的唯一来源。
5. Twitter/X、行情、Foundry、文件中转等外部能力必须明确真实适配器和最终状态来源，禁止在前端伪造完成状态。
6. 不得强行统一已经存在的特殊返回协议；修改接口时必须保持该业务当前前后端契约一致，并同步全部调用方。

## 六、数据库与后端门禁

1. 数据库结构修改必须新增 Flyway migration；禁止修改已经在任何环境执行过的历史 migration。
2. 禁止绕过 Flyway 直接在生产库手工修改结构；确需排障时必须先明确说明，并在代码中补齐对应 migration。
3. 数据库写操作必须检查真实影响行数；影响行数与合法请求数量不一致时，记录警告并返回失败。
4. 后端业务操作必须记录结构化日志：
   - 操作名称。
   - 关键业务 ID。
   - 请求数量。
   - 实际影响行数或返回数量。
   - 明确的成功或失败状态。
5. 批量增删改查必须能够从日志确认整批 ID、筛选条件和执行结果。
6. 日志禁止记录密钥、密码、令牌、完整 Prompt、文件内容、完整请求体及其他敏感数据。
7. 数据库、物理文件和外部服务共同参与的操作，任何一步失败都不得返回整体成功；必须保留明确错误和可定位日志。

## 七、UI 门禁

1. 所有搜索输入必须复用 `AIProvider-front/src/UiSearchField.jsx`。
2. 搜索图标固定在输入框右侧；实时筛选不得增加无意义的“搜索”提交按钮。
3. 禁止为单个页面另造搜索视觉、图标位置或交互方式。
4. 所有用户可见的一级工作区必须消费 `uiTheme` 语义变量；页面背景、卡片、正文、弱文字、边框和主按钮禁止直接写功能页私有颜色。
5. 仅安全敏感且已有注释说明的隔离弹窗允许使用固定高对比色。
6. 可点击元素必须使用原生交互元素；禁止 clickable `div`，禁止交互元素嵌套。
7. 自定义复合控件必须具备键盘操作、可见焦点和准确的无障碍名称。
8. 涉及搜索、主题、导航、响应式或交互语义的改动，必须更新或通过 `AIProvider-front/src/uiGate.test.js`；门禁失败禁止发布。

## 八、图片队列 CRUD 门禁

1. 本机图片、待处理、我的资产和回收站中的每张图片必须保留后端数据库分配的稳定数值 ID；前端分页 item 必须持有对应 ID。
2. 四个图片队列的新增、删除、修改、查询和队列迁移必须以数据库 ID 为业务主键，禁止使用文件路径或路径哈希代替 ID 执行数据库 CRUD。
3. 批量增删改必须由前端一次提交去重后的 `ids` 列表，后端使用一条批量 SQL 执行；禁止前端或后端逐条循环提交请求。
4. 后端根据 ID 查询数据库记录并取得文件路径；文件路径只用于交给本机 Bridge 执行读取、复制、移动或物理删除。
5. Bridge 不判断数据库记录是否存在，不维护服务器图片队列，不根据路径决定前端应移除哪个 item。
6. 后端实际影响行数必须等于本次合法去重 ID 数量，否则整个业务操作失败；前端禁止提示成功或擅自移除队列项。

## 九、Codex、测试与 Git

1. Codex 只实施主人已经确认的明确修改，不负责自行盘点项目、设计需求、寻找下一项任务或扩大检查范围。
2. 除非任务明确要求，不新增测试、不运行全量测试、不执行与本次修改无关的检查。
3. 只运行本次改动覆盖的最小业务测试、契约测试和对应门禁。
4. 所有测试只允许在本地工作区运行；生产服务器禁止运行测试。
5. 开发只使用 GitHub `master`，禁止创建或推送额外功能分支。
6. 每次完成修改后提交并推送 GitHub；提交前必须同步最新 `master`。
7. 并行任务不得同时修改公共协议、公共 DTO、数据库 migration、`AGENTS.md`、`releaseVersion.js` 或同一关键入口文件。
8. 发生冲突或发现远端已有相关修改时立即停止，禁止自动覆盖、强推或自行合并。
9. 并行任务不修改版本号；版本由最终发布或集成任务统一更新。

## 十、发布版本门禁

1. 首页左下角固定显示两行版本号，上面为“前端”，下面为“后端”。
2. 版本格式为 `年.月.日.当日发布序号.历史发布总序号`，例如 `2026.7.16.1.80`。
3. 前端和后端分别独立计数；只递增本次实际发布的一端，未发布的一端不得变化。
4. 每次生产发布前更新 `AIProvider-front/src/releaseVersion.js` 中对应端版本。
5. 跨日后当日发布序号从 1 重新开始，历史发布总序号继续累加。
6. 普通开发提交不因代码变化自动递增版本号；只有实际生产发布任务更新发布版本。

## 十一、生产发布

1. 当前生产配置的唯一权威来源是 `server-configs/production.aws.json`。
2. 发布前必须读取该文件，禁止在任务中重新猜测或硬编码主机、用户、私钥路径、源码目录、构建目录、服务名、运行目录、环境文件和健康检查地址。
3. `AGENTS.md` 不重复保存这些易变化的具体值；配置变化只修改对应 `production.<provider>.json`。
4. 发布必须基于明确的 `master` 提交哈希，服务器只拉取指定提交并完成：
   - 前端构建。
   - 后端打包。
   - 运行文件切换。
   - 服务重启。
   - 发布后健康检查。
5. 生产服务器禁止运行业务测试、契约测试或 UI 门禁。
6. 发布前验证若受本机旧 Bridge 进程干扰，可以停止本机 Bridge；停止 Bridge 不是 AWS 生产发布步骤。
7. 生产环境变量只允许读取任务所需字段，禁止在日志、回复、命令回显或提交中输出密钥和密码。
8. 当前生产不保留成功发布包备份；发布结束后必须确认发布备份目录为空。
9. 将来迁移到其他服务器时，新增 `server-configs/production.<provider>.json`，字段结构与当前 AWS 配置保持一致，并明确更新当前生效配置。

## 十二、构建产物清理

1. 生产发布成功并完成健康检查后，立即删除本次提交对应的临时构建目录及其中全部构建产物。
2. 发布失败时，只能在本次排障期间临时保留该提交对应的明确构建目录；排障结束后必须删除。
3. 本地不生成或长期保存生产发布包。
4. 历史流程或异常操作产生的 `.jar`、`.7z`、`.tar.gz` 或校验文件，确认不再用于当前排障后必须清理。
5. 删除前逐一核对目标是明确的构建目录或文件；禁止使用宽泛目录、未解析变量或递归通配删除。
