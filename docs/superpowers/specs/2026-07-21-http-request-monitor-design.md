# AIProvider HTTP 服务请求监控设计

## 背景与目标

监控中心当前“服务请求”卡片查询 `maid_LlmCallLogs`。该表表示 AI_maid 的 LLM 调用，不表示 AIProvider 后端请求；生产数据最后更新时间停在 2026-07-16，导致今日和最近 24 小时指标全部为 0。

目标是让该区域统计 AIProvider 后端收到的真实业务 API 请求，并提供今日请求总数、成功率、失败请求数、P95 响应时间和最近 24 小时趋势。

## 数据采集

新增后端 HTTP 请求监控拦截器，在请求结束后记录一条轻量指标：请求发生时间、规范化接口路径、HTTP 方法、响应状态码和耗时毫秒。记录不包含请求体、响应体、查询参数、Cookie、Token、提示词或文件内容。

以下请求不计入业务请求：

- `/api/monitor/**`，避免监控页面轮询形成自增长。
- `/api/health`，避免健康检查形成噪声。
- 非 `/api/**` 路径和静态资源。
- CORS 预检 `OPTIONS`。

异步、逐条或失败后降级到内存均不采用。拦截器通过一次数据库插入持久化当前请求；插入失败记录结构化错误日志，不伪造成功数据。

## 存储与查询

新增 Flyway 表 `c_HttpRequestMetrics`，使用数值自增 ID，并为请求时间及状态码建立监控查询索引。监控聚合查询改为读取该表：

- 成功：HTTP 200–399。
- 失败：其他已完成响应状态。
- 今日：按 `Asia/Shanghai` 的当天起点计算。
- 趋势：最近 24 小时按小时聚合。
- P95：只统计有耗时的已记录请求。

API 响应字段保持不变，因此前端图表与卡片无需改动。原 `maid_LlmCallLogs` 数据和 Maid/LLM 页面不变。

## 日志与保留

请求指标写入记录结构化业务日志，包含操作名、指标 ID、请求数量 1 和实际影响行数；影响行数不是 1 时记录警告并视为记录失败。清理沿用监控数据保留任务，按明确时间条件批量删除，并记录请求范围与实际影响行数。

## 验证范围

- 拦截器测试：业务 API 被记录；监控、健康检查、静态资源和 `OPTIONS` 不记录；状态码与耗时正确。
- Mapper/服务测试：今日汇总、成功率、失败数、P95 和 24 小时趋势从新表读取。
- 只运行新增/修改的后端测试及必要编译，不运行无关全量测试。
